= DDON Binary Analyzer
:toc:
:toclevels: 3

PyGhidra-based tooling for automated analysis of Dragon's Dogma Online PS4 binaries.
Extracts DWARF debug symbols and decompiles functions using Ghidra's headless analysis capabilities.

== Overview

This package provides two analysis modes for reverse engineering DDON binary formats:

* **DWARF Symbol Extraction** - Parses debug symbols to generate C++ struct definitions with dependency resolution
* **Function Decompilation** - Recursively analyzes function call graphs and extracts decompiled C code

Both modes operate through a unified Python entry point that interfaces with Ghidra via PyGhidra.

* **Separation of concerns** - Analysis logic, services, and output formatting are independent

== Usage

=== Command-Line Interface

The `run_ghidra_analysis.sh` wrapper script provides the primary interface:

[source,bash]
----
./run_ghidra_analysis.sh MODE TARGET BINARY [LOG_LEVEL] [DEPTH]
----

**Arguments:**

* `MODE` - Analysis mode: `dwarf` or `function`
* `TARGET` - Symbol name (DWARF mode) or qualified function name (function mode)
* `BINARY` - Program name in Ghidra project (e.g., `DDOORBIS.elf`)
* `LOG_LEVEL` - Optional: `INFO` (default) or `DEBUG`
* `DEPTH` - Optional: Recursion depth 0-10 (default: 2 for function, 5 for DWARF)

**Examples:**

[source,bash]
----
# Extract struct definitions for rLandInfo
./run_ghidra_analysis.sh dwarf "rLandInfo" "DDOORBIS.elf" "INFO" 2

# Decompile rLandInfo::load with call graph depth 3
./run_ghidra_analysis.sh function "rLandInfo::load" "DDOORBIS.elf" "INFO" 3
----

=== Direct Python Usage

[source,python]
----
from function_analyzer import FunctionDecompilationAnalyzer
from symbol_explorer import DwarfSymbolExplorer

# Function analysis
analyzer = FunctionDecompilationAnalyzer(program, "INFO")
analyzer.analyze_function("rLandInfo::load", max_depth=3)

# Symbol extraction
explorer = DwarfSymbolExplorer(program, "INFO")
explorer.analyze_symbol("rLandInfo", explore_depth=5)
----

== Analysis Modes

=== DWARF Symbol Extraction

Extracts C++ type definitions from DWARF debug information with full dependency resolution.

**Capabilities:**

* Structure and union definitions
* Template parameter extraction
* Inheritance chain resolution
* Nested type dependency tracking
* Bitfield handling
* Virtual table pointer identification

**Output Format:**

[source,cpp]
----
// Nested type definitions:

struct MtObject {
  _func_int * * _vptr$MtObject;  // offset: 0, size: 8
};

struct rLandInfo : public cResource {
  MtTypedArray<rLandInfo::cLandInfo> mLandInfo;  // offset: 112, size: 32
};
----

**Algorithm:**

1. Resolve symbol in DWARF category paths (`/DWARF/_UNCATEGORIZED_`, `/DWARF`)
2. Collect type dependencies using breadth-first traversal
3. Generate definitions in topological order (base classes first)
4. Format output with C++ syntax and size annotations

=== Function Decompilation

Recursively analyzes function implementations and their call graphs.

**Capabilities:**

* Decompiled C code extraction via `DecompInterface`
* Cross-reference analysis using `ReferenceManager`
* Struct-related function filtering by namespace
* Thunk resolution with `Function.getThunkedFunction()`
* Call graph traversal with configurable depth
* Duplicate elimination across recursion levels

**Output Format:**

[source,c]
----
// Function: load
bool __thiscall rLandInfo::load(rLandInfo *this, MtStream *in)
{
  // Decompiled implementation
}
----

**Algorithm:**

1. Resolve function using symbol table lookup with namespace awareness
2. Decompile using `DecompInterface` with timeout handling
3. Extract called functions using `Function.getCalledFunctions()`
4. Filter by namespace to identify struct-related calls
5. Recursively analyze struct-related functions up to max depth
6. Deduplicate by function signature hash

== Ghidra API Integration

=== Key API Methods Used

**Symbol Resolution:**

* `DataTypeManager.getDataType(CategoryPath, String)` - Direct type lookup
* `DataTypeManager.getAllComposites()` - Efficient composite iteration
* `Category.getDataTypesByBaseName(String)` - Conflict suffix handling
* `SymbolTable.getLabelOrFunctionSymbols(String, Namespace)` - Namespace-aware search

**Function Analysis:**

* `Function.getCalledFunctions(TaskMonitor)` - Call graph extraction
* `Function.getCallingFunctions(TaskMonitor)` - Reverse call analysis
* `Function.getThunkedFunction(boolean)` - Thunk chain resolution
* `DecompInterface.decompileFunction(Function, int, TaskMonitor)` - Decompilation

**Type Analysis:**

* `Composite.getDefinedComponents()` - Field iteration (excludes padding)
* `DataType.getParents()` - Reverse dependency tracking
* `Namespace.getPathList(boolean)` - Namespace path extraction
* `Structure.getDataTypeAt(int)` - Offset-based component lookup

=== Optimization Strategies

**Caching:**

* Type cache populated using specialized iterators (`getAllComposites()` before `getAllDataTypes()`)
* Symbol lookups cached by full path and simple name
* Namespace resolution memoized during function search

**Efficient Iteration:**

* Use `getAllComposites()` instead of filtering `getAllDataTypes()`
* Leverage `CategoryPath` direct lookups before cache population
* Wildcard searches via `findDataTypes()` for pattern matching

**Error Handling:**

* Timeout configuration for decompilation (default: 30s)
* Graceful degradation when symbols unavailable
* Validation of `DecompileResults` state before accessing

== Output Formats

=== DWARF Mode

Generates C++ header-style definitions:

[source,cpp]
----
// Nested type definitions:
struct BaseType { /* fields */ };

// Main type definition:
struct DerivedType : public BaseType { /* fields */ };
----

**Metadata included:**

* Offset and size for each field
* Pack value for structures
* Total structure size
* Inheritance relationships

=== Function Mode

**INFO Level:**

Decompiled code only, no metadata:

[source,c]
----
bool __thiscall rLandInfo::load(rLandInfo *this, MtStream *in)
{
  // Implementation
}
----

**DEBUG Level:**

Full analysis with metadata, cross-references, and call graphs.

== Implementation Notes

=== Symbol Search Strategy

1. **Direct CategoryPath lookup** - Most efficient for known DWARF paths
2. **Wildcard search** - Pattern matching via `findDataTypes()` if wildcards present
3. **Cache lookup** - Fallback to pre-populated cache with priority sorting
4. **Conflict resolution** - `getDataTypesByBaseName()` for `.conflict` suffixes

Priority order: DWARF composites > non-DWARF composites > DWARF types > other types

=== Function Resolution Strategy

1. **Symbol table iterator** - `getSymbolIterator(name, caseSensitive)`
2. **Namespace-aware lookup** - `getLabelOrFunctionSymbols(name, namespace)` for qualified names
3. **Exact match** - Function manager iteration with name comparison
4. **Partial match** - Token-based matching as fallback
5. **Thunk resolution** - Automatic resolution to target function

=== Dependency Collection

**Breadth-first traversal:**

1. Start with root type
2. Extract component types from composites
3. Resolve typedefs to concrete types
4. Track visited types to prevent cycles
5. Build dependency graph for topological sorting

**Base class handling:**

* Detect via first component named `super_*` at offset 0
* Unwrap typedefs to get actual base structure
* Ensure base classes precede derived in output order

== Troubleshooting

**Symbol not found:**

* Verify symbol exists in DWARF debug info
* Check category paths: `/DWARF/_UNCATEGORIZED_/`, `/DWARF/`
* Try wildcard search: `*SymbolName*`
