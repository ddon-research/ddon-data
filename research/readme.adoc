This guide explains how to decode resource files from Dragon's Dogma Online client data using PS4 debug symbols and ImHex pattern definitions.

== Prerequisites

Before starting, ensure you have:

* Basic knowledge of C/C++ programming concepts
* Understanding of reverse engineering fundamentals  
* https://imhex.werwolv.net/[ImHex] hex editor installed
* Access to PS4 IDA dumps in the `binaries/` folder of this repository

=== For Automated Analysis (Recommended)

* **PyGhidra** - Python interface for headless Ghidra execution
* **Ghidra 11.4+** - With PS4 DDON project already imported
* **Configured `.env` file** - Located in `research/ghidra/` with paths and settings
* **Bash environment** - WSL, Git Bash, or native Linux/macOS for running shell scripts

== Getting Started

=== 1. Identify Target Resource

Check unsupported extensions in https://github.com/ddon-research/ddon-extractor/blob/develop/docs/deserialization/supported-file-extensions.csv[ddon-extractor documentation] and compare with resource dumps (e.g., `03040008-resources.csv`).

=== 2. Extract Resource Structure  

Use Ghidra automation to get struct definitions and load function analysis:

[source,bash]
----
cd research/ghidra
# Get struct definitions
./run_ghidra_analysis.sh dwarf "rResourceName" "DDOORBIS.elf" > struct_output.txt

# Get load function with magic numbers
./run_ghidra_analysis.sh function "rResourceName::load" "DDOORBIS.elf" > function_output.txt
----

=== 3. Create ImHex Pattern

Convert Ghidra output to ImHex pattern format in `research/<version>/rResourceName-pattern.cpp`.

=== 4. Test and Iterate

Validate patterns with ImHex CLI and adapt for Season 3 differences:

[source,bash]
----
# Test pattern
imhex --pl format --pattern rResourceName-pattern.cpp --input resource.ext

# Batch validation 
./validate_pattern.sh rResourceName-pattern.cpp test_files/ "field1,field2"
----

== Practical Example: rLandInfo

This demonstrates the complete workflow using `rLandInfo`, which has structural differences between Season 2 and Season 3.

=== Struct Analysis with Ghidra

Running the DWARF symbol explorer on `rLandInfo`:

[source,bash]
----
./run_ghidra_analysis.sh dwarf "rLandInfo" "DDOORBIS.elf"
----

**Output (key structures)**:
[source,cpp]
----
struct rLandInfo : public cResource {
  MtTypedArray<rLandInfo::cLandInfo> mLandInfo;
};

struct rLandInfo::cLandInfo : public MtObject {
  u32 mLandId;
  MtTypedArray<rLandInfo::cLandInfo::cLandAreaInfo> mAreaArray;
};

struct rLandInfo::cLandInfo::cLandAreaInfo : public MtObject {
  u32 mAreaId;
};
----

The structure shows a hierarchical array: lands contain arrays of area info objects.

=== Load Function Analysis

Running the function analyzer reveals the binary format:

[source,bash]
----
./run_ghidra_analysis.sh function "rLandInfo::load" "DDOORBIS.elf"
----

**Key findings**:
- Magic number: `4800844` = `0x4C414900` = `"LAI\0"`  
- Season 2 version: `2`, Season 3 version: `4`
- Binary layout: magic → version → array data

=== ImHex Pattern Development

**Season 2 Base Pattern**:
[source,cpp]
----
struct MtObject { };

struct cResource {
    char magicString[4];
    u32 magicVersion;
};

struct rLandInfo_cLandInfo_cLandAreaInfo : MtObject {
    u32 mAreaId;
};

struct rLandInfo_cLandInfo : MtObject {
    u32 mLandId;
    MtTypedArray<rLandInfo_cLandInfo_cLandAreaInfo> mAreaArray;
};

struct rLandInfo : cResource {
    MtTypedArray<rLandInfo_cLandInfo> mLandInfo;
};

rLandInfo rlandinfo_at_0x00 @0x00;
----

**Season 3 Adaptation**:
Through iterative testing, Season 3 adds two fields to `cLandInfo`:
[source,cpp]
----
struct rLandInfo_cLandInfo : MtObject {
    u32 mLandId;
    bool mIsDispNews;    // New in Season 3
    u8 mGameMode;        // New in Season 3  
    MtTypedArray<rLandInfo_cLandInfo_cLandAreaInfo> mAreaArray;
};
----

== Complete Workflow Summary

The complete automated pipeline combines analysis, pattern generation, and validation:

[source,bash]
----
# 1. Extract information from debug symbols
./run_ghidra_analysis.sh dwarf "rResourceName" "DDOORBIS.elf" > struct_output.txt
./run_ghidra_analysis.sh function "rResourceName::load" "DDOORBIS.elf" > function_output.txt

# 2. Create ImHex pattern from output
# (Convert struct definitions to pattern format in research/<version>/)

# 3. Validate and iterate  
imhex --pl format --pattern rResourceName-pattern.cpp --input season2_resource.ext
imhex --pl format --pattern rResourceName-pattern.cpp --input season3_resource.ext

# 4. Batch validation
./validate_pattern.sh rResourceName-pattern.cpp test_files/ "field1,field2"
----

**Iteration Process:**
1. If Season 3 validation fails, analyze error patterns  
2. Adjust pattern incrementally (add/remove fields)
3. Re-validate with ImHex CLI for quick feedback
4. Document structural differences for future reference

== Ghidra Headless Analysis Scripts

The repository includes PyGhidra automation scripts for extracting C++ definitions and function analysis from DWARF debug symbols:

[source,bash]
----
./run_ghidra_analysis.sh <mode> <symbol/function> <program>
----

**Modes:**
- `dwarf` - Extract struct definitions with inheritance and dependencies
- `function` - Decompile functions with call graph analysis

**Usage:**
[source,bash]
----
cd research/ghidra

# Extract struct definitions
./run_ghidra_analysis.sh dwarf "rResourceName" "DDOORBIS.elf"

# Analyze load functions  
./run_ghidra_analysis.sh function "rResourceName::load" "DDOORBIS.elf"

# Quiet mode (logs only)
./run_ghidra_analysis.sh dwarf "rResourceName" "DDOORBIS.elf" --quiet
----

**Configuration:** Scripts read from `.env` file:
- `PYGHIDRA_RUN` - PyGhidra executable path
- `PROJECT_DIR`/`PROJECT_NAME` - Ghidra project location
- `DEFAULT_LOG_LEVEL`/`DEFAULT_EXPLORE_DEPTH` - Analysis parameters

**Output:** Clean C++ code with proper formatting, inheritance syntax, and dependency ordering.

== ImHex Command Line Interface

ImHex CLI enables automated pattern testing and batch validation:

[source,bash]
----
imhex --pl format --pattern <pattern_file> --input <resource_file>
----

**Key Options:**
- `--pl format` - Pattern language formatter plugin
- `--pattern <file>` - ImHex pattern file (.cpp or .hexpat)  
- `--input <file>` - Binary resource file to decode
- `--output <file>` - JSON output file (optional, defaults to stdout)

**Usage:**
[source,bash]
----
# Test pattern
imhex --pl format --pattern rLandInfo-pattern.cpp --input land_list.lai

# Batch validation with custom script
./validate_pattern.sh rLandInfo-pattern.cpp test_files/ "mLandId,mGameMode"
----

**Output:** Valid JSON on success, error messages on failure (e.g., `Pattern Error: runtime error: Array grew past set limit`).
