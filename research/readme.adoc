Systematic workflow for decoding DDON resource files using PS4 debug symbo automation. Designed for LLM-assisted reverse engineering.

== Prerequisites

=== Required Tools
* **ImHex** - Hex editor with pattern language support (CLI enabled)
* **PyGhidra** - Headless Ghidra automation
* **Ghidra 11.4+** - With PS4 DDON project imported
* **Bash environment** - WSL, Git Bash, or native Linux/macOS for automation scripts

=== Required Data  
* **PS4 Debug Symbols** - DWARF symbols imported in Ghidra project
* **Season 2 Resource Files** - Located in `DDON_02030004/nativePC/rom/`
* **Season 3 Resource Files** - Located in `DDON_03040008/nativePC/rom/` 
* **Configuration** - `.env` file in `research/ghidra/headless-scripts/` with paths and settings

== Automation Workflow

=== Step 1: Resource Identification

**Objective**: Identify target resource type for reverse engineering.

**LLM Automation Prompt**:
```
Given a file extension `.ext`, help me identify if it needs ImHex pattern development by:
1. Checking if it exists in ddon-extractor supported extensions CSV
2. Looking for the extension in resource dumps (e.g., 03040008-resources.csv)  
3. Determining the likely resource class name (usually `r<Name>Info` format)
```

**Commands**:
```bash
# Check extension support status
grep "\.ext" ddon-extractor/docs/deserialization/supported-file-extensions.csv

# Find resource instances  
grep "\.ext" client/03040008-resources.csv | head -5
```

=== Step 2: Symbol Extraction

**Objective**: Extract C++ struct definitions and load function analysis from debug symbols.

**LLM Automation Prompt**:
```
Help me extract DWARF symbols for resource type `rResourceName`. Execute these commands and provide the outputs:

1. Extract struct definitions: `./run_ghidra_analysis.sh dwarf "rResourceName" "DDOORBIS.elf"`
2. Extract load function: `./run_ghidra_analysis.sh function "rResourceName::load" "DDOORBIS.elf"`

Analyze the outputs to identify:
- Main struct hierarchy and inheritance relationships
- Magic number (hex constant in load function)  
- Version number (integer comparison in load function)
- Array structures and nested types
```

**Commands**:
```bash
cd research/ghidra/headless-scripts

# Get complete struct definitions with dependencies
./run_ghidra_analysis.sh dwarf "rResourceName" "DDOORBIS.elf" > ../../rResourceName.h

# Get load function with magic numbers and binary layout
./run_ghidra_analysis.sh function "rResourceName::load" "DDOORBIS.elf" > ../../rResourceName_load.cpp
```

=== Step 3: Pattern Generation

**Objective**: Convert C++ struct definitions to ImHex pattern format.

**LLM Automation Prompt**:
```
Convert the extracted C++ structs and load function to ImHex pattern format. 

**Input Files**:
- `rResourceName.h` - Struct definitions from DWARF extraction
- `rResourceName_load.cpp` - Load function with magic numbers

**Conversion Rules**:
1. **Base Types**: Use standard DDON base structures:
   ```cpp
   struct MtObject {};
   struct cResource { char magicString[4]; u32 magicVersion; };
   struct MtArray { u32 mLength; };
   template<typename T> struct MtTypedArray : MtArray { T arr[mLength]; };
   ```

2. **Namespace Flattening**: Convert `rResourceName::cClassName` → `rResourceName_cClassName`

3. **Type Mapping**: 
   - `uint32_t` → `u32`, `int32_t` → `s32`, `uint64_t` → `u64`
   - Skip `_vptr$ClassName` virtual table pointers
   - Skip padding fields unless functionally significant

4. **Magic Number Extraction**: From load function hex constants (e.g., `0x49414c` → `"LAI\0"`)

5. **Template Instantiation**: Convert templated types to concrete implementations

**Output**: Complete `.cpp` pattern file ready for ImHex CLI testing.
```

**Output Location**: `research/02030004/rResourceName-pattern.cpp`

=== Step 4: Validation and Cross-Season Testing

**Objective**: Validate Season 2 pattern and identify Season 3 differences.

**LLM Automation Prompt**:
```
Help me validate the ImHex pattern through systematic testing:

1. **Baseline Test**: Validate Season 2 pattern with Season 2 data
2. **Cross-Season Test**: Test Season 2 pattern on Season 3 data to identify breaks
3. **Error Analysis**: Interpret failure modes and suggest investigation approaches

Execute these validation commands and analyze results:
```

**Commands**:
```bash
# Baseline validation (should succeed)
imhex --pl format --pattern 02030004/rResourceName-pattern.cpp --input /path/to/DDON_02030004/.../resource.ext

# Cross-season validation (may fail)  
imhex --pl format --pattern 02030004/rResourceName-pattern.cpp --input /path/to/DDON_03040008/.../resource.ext
```

**Expected Outcomes**:
- **✅ Baseline Success**: JSON output with correct field values
- **❌ Cross-Season Failure**: Error patterns indicating struct changes

=== Step 5: Season 3 Adaptation (Manual Process)

**Objective**: Create Season 3 pattern through iterative reverse engineering.

**LLM Guidance Prompt**:
```
The Season 2 pattern failed on Season 3 data. Help me systematically identify structural changes:

1. **Error Analysis**: What does the failure mode indicate?
   - "Array grew past set limit" → Field misalignment, not array size
   - "Parsing stops early" → Missing or incorrectly sized fields  
   - Version mismatch → Magic number differences

2. **Investigation Strategy**: Guide hex comparison between Season 2 and Season 3 files
3. **Incremental Testing**: Suggest field additions/modifications to test
4. **Pattern Updates**: Help create `research/03040008/rResourceName-pattern.cpp`
```

**Process**:
1. Compare hex data between seasons manually
2. Hypothesize new field locations and types
3. Update pattern incrementally  
4. Test each change with ImHex CLI
5. Document discovered differences

== Common Error Patterns

**Array Limit Errors**:
```
Pattern Error: runtime error: Array grew past set limit of 65536
```
**Cause**: Field misalignment, not array size. Add `#pragma array_limit <number>` only after confirming array size is actually large.

**Version Mismatch**:
```
Pattern fails silently or returns incorrect data
```
**Cause**: Magic version number changed between seasons. Update `magicVersion` field in pattern.

**Parsing Stops Early**:
```
JSON output truncated or missing expected fields
```
**Cause**: New fields added between debug symbols and Season 3. Manual field discovery required.
